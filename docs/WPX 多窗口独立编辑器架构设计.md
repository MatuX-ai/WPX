# WPX 多窗口独立编辑器架构设计

**版本**：V1.1（已实现）

## 1. 概述

WPX 是一个基于 Electron + Vue3 的 AI 智能文档编辑器。多窗口独立编辑器架构已于 **V1.0 全面实现并通过验收**（2026-06-23，73/73 检查点 100% 通过），使每个文档拥有独立的编辑窗口，同时共享资料库、记忆库、智能模板等全局资源。

**实施结论**：所有设计文档约定的功能均已实现，所有 P0 安全漏洞已修复。详细验收结果见 [多窗口架构验收报告](./多窗口架构验收报告.md)。

**核心原则**：每个窗口是独立的编辑上下文，但用户身份、偏好和知识资产全局唯一。

---

## 2. 设计目标

- **独立编辑**：每个窗口打开一个独立文档，拥有完整的编辑器状态（内容、撤销栈、AI 对话历史）。

- **共享全局数据**：资料库、智能模板、用户偏好、主题等全局共享，一处更新所有窗口生效。

- **原生体验**：支持 Cmd/Ctrl+N 新建窗口，窗口列表管理，关闭时保存提示，最后一个窗口关闭时不退出应用（符合 macOS 习惯）或可选择行为。

- **最小化架构侵入**：不影响现有单窗口的渲染逻辑，通过参数驱动复用同一套 Vue 前端代码。

- **性能可控**：限制最大窗口数量（默认 8 个），避免内存溢出；非活跃窗口可降低资源占用。

### 3.1 整体架构图

### 3.2 主进程窗口管理器 `WindowManager`)

新建 `electron/window-manager.js`，职责：

- 维护一个窗口池 `Map<windowId, BrowserWindow>`。

- 统一创建窗口：接受 `docPath` 可选参数，新窗口加载同一入口 `dist/index.html`，通过 query string 传递 `windowId` 和初始文档路径。

- 管理窗口间消息转发：渲染进程需要广播时，由主进程中转。

- 监控窗口数量，超过上限时提示用户关闭旧窗口。

- 处理窗口关闭事件：通知渲染进程进行保存检查，执行关闭或取消。

### 3.3 渲染进程适配

无需创建多个前端入口，所有窗口加载同一套 Vue 应用，通过路由参数或 store 初始化时识别自身 `windowId`。

- 在 `App.vue` 或 `EditorLayout.vue` 的 `setup` 中，从 `window.__WPX_WINDOW_ID__` 读取当前窗口 ID（由 preload 脚本注入或从 URL 参数解析）。

- 根据窗口 ID，从主进程获取该窗口绑定的文档数据（或新建文档）。

- 每个窗口拥有独立的 Pinia store 实例（Vue 应用中 store 默认是单例，但由于是不同渲染进程，天然隔离）。

## 4. 窗口生命周期管理

### 4.1 新建窗口

触发方式：

- 全局快捷键 `Ctrl+N`（主进程直接创建新窗口，加载空白文档）。

- 菜单栏「文件 → 新建窗口」。

- 托盘菜单「新建文档」。

主进程逻辑：

1. `WindowManager.createWindow()` 生成一个新的 `BrowserWindow`，分配唯一 `windowId`。

2. 窗口加载 `file://dist/index.html#/editor?windowId=xxx`[。](file://dist/index.html#/editor?windowId=xxx`。)

3. 渲染进程初始化时，检测到无绑定的文档路径，则创建全新空白文档。

4. 新建窗口聚焦到最前。

### 4.2 窗口聚焦与切换

- 用户可通过系统任务栏或应用内「窗口」菜单切换。

- 多窗口列表菜单项动态生成，由主进程收集当前所有窗口标题并通过 IPC 推送给各个窗口的 TitleBar 组件（或仅在主进程菜单中显示）。

- 点击菜单项时，主进程调用 `BrowserWindow.focus()`。

### 4.3 关闭窗口与保存逻辑

这是最关键的用户体验环节：

1. 用户点击窗口关闭按钮（我们的自定义 TitleBar 按钮）或按 Cmd/Ctrl+W。

2. 渲染进程检测到关闭意图后，检查文档是否有未保存修改。

3. 若有修改：

   - 弹出模态对话框：“是否保存对「文档名」的更改？”，提供「保存」「不保存」「取消」。

   - 「保存」：触发保存流程（自动触发智能文库保存对话框），完成后通知主进程关闭窗口。

   - 「不保存」：直接通知主进程关闭窗口。

   - 「取消」：什么都不做，窗口继续打开。

4. 若无修改，直接通知主进程关闭窗口。

**实现注意**：

- 渲染进程通过 IPC `send('request-close-window')` 到主进程，主进程不能直接关闭，需等待渲染进程的后续指令。

- 可以复用已有的 `SaveDialog.vue` 组件。

- 为防止窗口关闭时浏览器上下文丢失导致未保存数据，主进程应设置 `BrowserWindow` 的 `close` 事件，并调用 `event.preventDefault()`，等待渲染进程告知后再销毁窗口。

### 4.4 最后一个窗口关闭行为

提供配置项（默认遵循操作系统惯例）：

- **macOS**：最后一个窗口关闭时，应用不退出，保持程序运行（托盘或 Dock 中可见）。

- **Windows/Linux**：最后一个窗口关闭时，默认退出应用，但可在设置中改为最小化到托盘。

主进程维护窗口计数，当窗口数从 1 变为 0 时，根据平台和设置决定 `app.quit()` 或仅隐藏。

## 5. 跨窗口数据共享

### 5.1 用户偏好与主题

- 用户偏好（主题、默认字体、语言等）存储在主进程的 `electron-store` 或本地 JSON 文件中。

- 渲染进程启动时，通过 IPC `get-user-preferences` 获取初始值，并监听主进程的 `preferences-changed` 广播事件，实时更新。

- 当用户在任一窗口修改偏好，渲染进程发送 `update-preferences` 到主进程，主进程写入存储并广播给所有窗口。

### 5.2 资料库与记忆库

- 资料库（用户上传的参考文档索引、向量存储）和记忆库（用户习惯、智能模板）原本计划在渲染进程或后端服务中。多窗口下，必须统一抽离到主进程或独立本地服务。

- **资料库索引**：可以使用一个轻量的本地数据库（如 `better-sqlite3` 或 `lowdb`）在主进程中管理，渲染进程通过 IPC 进行 CRUD 操作。当任意窗口上传资料或修改资料，主进程更新存储后通知其他窗口刷新侧边栏列表。

- **记忆库/智能模板**：同样放在主进程存储。用户在某窗口的每次格式调整，渲染进程发送到主进程汇总，主进程进行习惯分析和模板生成，并广播结果。

### 5.3 实时同步视图

- 资料库侧边栏`KnowledgePanel.vue`）和智能模板列表，在每个窗口都展示且应该保持一致。通过主进程的广播机制实现：

  - 资料库变化 → 主进程发送 `knowledge-updated` 事件 → 所有渲染进程更新视图。

  - 模板变化 → 发送 `templates-updated`。

## 6. 跨窗口通信定义

所有通信基于 Electron IPC，不使用 `window.postMessage` 或共享 Web Worker。

### 6.1 IPC 通道约定

| 通道名称 | 方向 | 描述 |

|:---|:---|:---|

| `window:create` | 渲染→主 | 请求新建窗口（可选传递文档路径） |

| `window:request-close` | 渲染→主 | 窗口发起关闭请求，主进程暂不销毁窗口，等待下一步 |

| `window:confirm-close` | 渲染→主 | 渲染进程确认可以安全关闭窗口 |

| `window:cancel-close` | 渲染→主 | 取消关闭 |

| `window:focus` | 主→渲染 | 通知窗口它获得了焦点（用于激活时的UI刷新） |

| `data:preferences:get` | 渲染→主 | 获取用户偏好 |

| `data:preferences:set` | 渲染→主 | 更新偏好 |

| `data:preferences:changed` | 主→渲染（广播） | 偏好已更改 |

| `data:knowledge:updated` | 主→渲染（广播） | 资料库索引变化 |

| `data:templates:updated` | 主→渲染（广播） | 智能模板列表变化 |

| `data:memory:record` | 渲染→主 | 记录用户操作习惯 |

### 6.2 广播实现

主进程维护一个 `Set<webContents>`，包含所有活跃窗口的 webContents。需要广播时，遍历发送消息。新窗口创建时注册，窗口销毁时注销。

## 7. 与现有功能的集成

### 7.1 AI 助手

- 每个窗口有自己独立的 AI 对话历史，AI 头像和对话窗的浮层组件无需跨窗口通信。

- 但是，AI 调用的底层模型配置（API Key、模型选择）应从全局偏好中获取，修改后所有窗口即时生效。

### 7.2 全局快捷键

- 全局快捷键（如 `Ctrl+Shift+W` 呼出 AI 对话窗）现在需要明确作用域：这些快捷键应只对当前聚焦窗口有效，而非全局都触发同一窗口的动作。

- 主进程捕获全局快捷键后，通过 IPC 发送给当前聚焦窗口的 webContents。

- 需要新增一个 `shortcut:ai-chat-toggle` 的 IPC 事件，由当前活动窗口响应。

### 7.3 系统托盘

- 托盘逻辑依然在主进程，不受多窗口影响。

- 新增托盘菜单项：「新建窗口」「窗口列表」（子菜单动态显示所有窗口标题，点击切换到对应窗口）。

### 7.4 自定义标题栏

- TitleBar 组件需增加“窗口菜单”入口（图标），点击后通过 IPC 请求主进程显示窗口列表菜单。

- 关闭、最小化、最大化按钮调用 IPC，通知主进程操作当前窗口（通过 `windowId` 或 `event.sender` 识别）。

## 9. 潜在问题与应对

| 问题 | 应对策略 |

|:---|:---|

| **内存占用过高** | 限制最大窗口数（8个）；非活跃窗口可提示用户关闭；未来可考虑将非活跃渲染进程降级为 `backgroundThrottling`。 |

| **资料库并发写入冲突** | 所有数据操作由主进程代理，通过队列处理，避免多渲染进程直接写文件。 |

| **窗口ID管理** | 使用递增整数，不重用，通过 `electron-store` 持久化当前活跃窗口列表，防止重启后状态丢失（不必须）。 |

| **macOS 与 Windows 行为差异** | 通过 `process.platform` 在管理器中做平台判断，统一 API 接口。 |

| **开发调试复杂性** | 利用 `electron:dev` 脚本`--remote-debugging-port` 为每个窗口分配不同端口，方便 DevTools 单独调试。 |
